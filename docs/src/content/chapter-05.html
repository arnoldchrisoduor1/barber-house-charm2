What Haus of Grooming OS is

A multi-tenant SaaS for grooming, wellness, field services, therapy, solo independents, and retail product businesses. Each Haus (business mode) reuses one core domain — orgs, branches, staff, services, customers, bookings, transactions — and swaps labels, sidebar sections, and specialist screens (clinical, coverage zones, session notes, shop orders, etc.).

Implementation touchpoints: src/components/AppLayout.tsx (getNavForMode, mode nav constants), src/hooks/useBusinessCategory.tsx (MODE_TERMS, effectiveCategories, mobileSpecialty), src/hooks/useSubscription.ts (plan + hasFeature), src/lib/pricing.ts (SaaS pricing).

How money works (two different flows)

1) Tenant → platform (SaaS): Plans in src/lib/pricing.ts — solo_pro, starter, professional, enterprise — per platform (Barber, Beauty, … Products) and billing cycle. useSubscription reads subscriptions.plan; hasFeature unlocks Professional/Enterprise modules (e.g. pos_payments).

2) End-customer → tenant: In-venue and online take-home revenue via POS, M-Pesa, other tenders — modeled in transactions; UI on /pos, /reconciliation, /payments-demo where present. This is not the SaaS fee.

3) Staff inside the tenant: My Earnings, Tips, Commissions, Payroll (feature-gated) — routes like /my-earnings, /tips, /commissions, /payroll.

How bookings work

Shared schema: bookings, booking_services, links to customers, staff, services, branches (see Chapter 4). Statuses follow booking_status enum.

Labels change by Haus via MODE_TERMS — e.g. “Appointment” (barber), “Session” (spa), “Home Visit” (mobile), “Consultation” (clinic). For mobile with a mobileSpecialty, the UI can use the specialty’s terms for barber/beauty/etc. while keeping mobile in effectiveCategories for filtering.

Staff manage on /bookings and /schedule; customers use public book and /portal/bookings. Conflict rules and availability live in the API service layer in production.

Deep dive: Haus of Mobile

Nav source: MOBILE_NAV — dispatch (exec/manager): routes, home visits, /coverage-zones, /field-operations, team, clients. Field staff: My Hub with visits, schedule, zones, earnings, portfolio, WhatsApp, loyalty, packages, tips, QR clock, chat.

Terminology: MODE_TERMS.mobile uses “Home Visit” / “Service Area”; mobileSpecialty (barber, beauty, spa, nail_bar, clinic, therapy) drives borrowed labels via effectiveCategories for services/bookings.

Dashboard: When category is only mobile, DashboardPage.tsx renders MobileDispatchDashboard (management) or MobileSoloProDashboard (staff) — visit lists, online toggle, daily goal, demo route/kit UI (not used for solo_pro business type).

Deep dive: Haus of Solo Pro

Nav: No SOLO_PRO_NAV in code — getNavForMode falls back to BARBER_NAV (barber section names, same routes).

Onboarding: Solo Barber / Beauty / Spa / … via category=solo_pro&specialty=… on PlatformsPage / Navbar.

Terms & theme: MODE_TERMS.solo_pro (“Professional”, “Workspace”); theme theme-solo.

Pricing SKU: Plan solo_pro in pricing.ts (dedicated platform price).

Dashboard: Standard dashboard — not the mobile visit dispatch UI (that is reserved for mobile only).

Deep dive: Haus of Products

Nav: PRODUCTS_NAV — executive store analytics; Store Floor (POS, stock, /shop-orders, attendance); Till & Sales Floor for cashier/associate; Catalogue (inventory, consumption, suppliers, price lock); growth (loyalty, bundles, WhatsApp, product gallery).

Dashboard: ProductsDashboard when the only selected category is products (isProductsOnly in DashboardPage.tsx).
